4.2 Top-Level-Zerlegung

Das Framework ist als Maven-Bibliothek strukturiert — es läuft nicht selbst, sondern wird in Fachapplikationen eingebettet. Die Zerlegung folgt dem Prinzip Kern + optionale Erweiterungsmodule:

e-gov-bom                  ← Dependency-Versionssteuerung (kein Code)
e-gov-framework            ← Pflicht-Kern: Persistence, IAM, Command-Pattern, Exception-Handling
e-gov-luid                 ← Optional: alternative ID-Generierungsstrategien
e-gov-mdm                  ← Optional: Stammdatenverwaltung (Codelisten, Vokabulare)
e-gov-mdm-api              ← Optional: REST-API-Layer für MDM
e-gov-ai-rule-generation   ← Optional: LLM-gestützte Regelgenerierung
e-gov-framework-demo       ← Referenzimplementierung (nicht für Produktion)

4.3 Wesentliche Designprinzipien

  • Convention over Configuration — Spring Boot Auto-Konfiguration; consuming Apps benötigen nur wenige egov.*-Properties
  • Command Pattern für alle Schreiboperationen — jede fachliche Aktion wird als Command-Entity persistiert; ermöglicht Audit-Trail und nachvollziehbaren Rollback
  • Deklarative Autorisierung via Annotationen — @Ruleset, @AllowedForRoles, @Validation auf DTOs; Laufzeit-Auswertung durch ValidationEngine
  • Row-based Security by Default — QueryBuilder fügt automatisch Permission-Filter ein; kein explizites opt-in nötig
  • Eigene Filter-DSL — ermöglicht dynamische, ausdrucksstarke WHERE-Klauseln aus REST-Anfragen ohne SQL-Injection-Risiko
  • Soft-Delete — Archivierung statt physischem Löschen via ArchivedModifiableEntity
