3 Kontextabgrenzung

3.1 Fachlicher Kontext

Das e-gov-framework selbst hat keine fachliche Benutzeroberfläche. Es ist eine Bibliothek, die von Fachapplikationen verwendet wird.

Partner Eingehende Daten Ausgehende Daten
Sachbearbeiter (Behörde) Formulardaten, Filter- und Suchanfragen, CRUD-Operationen Gefilterte Datenlisten (nur berechtigte Zeilen), Validierungsfehler, Statusmeldungen
Behörden-IAM (Azure AD + F5) EGOV_TOKEN-Cookie mit userId —
Stammdaten-Konsumenten (MDM) Abfragen auf Codelisten (Ortschaften, Sprachen, Anreden etc.) Mehrsprachige Code-Listen

3.2 Technischer Kontext

Schnittstelle Protokoll Richtung Bemerkung
Azure AD + F5 Firewall HTTPS / Cookie → Framework Setzt EGOV_TOKEN-Cookie nach Login; Framework extrahiert userId über UserIdProvider-Interface
PostgreSQL JDBC ↔︎ Primäre Produktionsdatenbank; Framework liefert Flyway-DDL-Skripte
SQL Server JDBC ↔︎ Alternative für Kunden mit Microsoft-Stack
ZooKeeper TCP (Port 2181) ↔︎ Optional; für verteilte Command-Ausführung über mehrere Instanzen; via Apache Curator
Azure OpenAI (gpt-4o) HTTPS / REST → Nur e-gov-ai-rule-generation-Modul; generiert Validierungsregeln aus natürlicher Sprache
Artifactory (bin.ti8m.ch) HTTPS / Maven ← Framework-JARs werden von consuming Apps via Maven bezogen
ArgoCD Kubernetes API → Deployment-Steuerung der consuming Apps; kein direkter Framework-Kontakt
