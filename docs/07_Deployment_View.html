7 Verteilungssicht

7.1 Motivation

Das e-gov-framework wird vollständig On-Premise auf kundeneigenen Kubernetes-Clustern betrieben. Diese Entscheidung folgt aus behördlichen Anforderungen an Datensouveränität, Compliance und Netzwerkisolation. Alle Kernmodule benötigen ausschliesslich Infrastruktur innerhalb des Clusters (→ Randbedingungen Sektion 2). Das optionale KI-Modul e-gov-ai-rule-generation bildet die einzige Ausnahme: Es kommuniziert mit Azure OpenAI über HTTPS und erfordert daher eine explizite Netzwerk-Freigabe durch die Behörde (→ ADR-007).

│  ┌─ Azure AD (extern) ─┐    ┌──── Cloud-Egress (nur KI-Modul, optional) ─────────┐ │
│  │  Identity Provider  │    │  e-gov-ai-rule-generation → Azure OpenAI (HTTPS)   │ │
│  └─────────────────────┘    └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────────────────────┘

Sicherheitszonen: - Internet-Zone: Eingehender HTTPS-Traffic der Sachbearbeiter; TLS-Terminierung durch die F5-Firewall; EGOV_TOKEN-Cookie wird gesetzt - Cluster-Zone (On-Premise): Alle Framework-Kernmodule und Datenbankserver; kein direkter Internetzugang; Kommunikation nur Pod-zu-Pod und Pod-zu-DB - Cloud-Egress (optional): Ausschliesslich e-gov-ai-rule-generation; erfordert explizite Firewall-Freigabe durch die Behörde (→ ADR-007)
